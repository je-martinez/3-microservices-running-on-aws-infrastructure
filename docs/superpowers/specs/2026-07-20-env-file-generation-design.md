# Env-File Generation Design

## Summary

Auto-generate every env file that derives from Terraform discovery, split per consumer, so
`make env-file` produces them all and no value is hand-maintained. This is **block 3 of 3** of
the "Developer Experience" milestone — block 1 (scripts to Python) and block 2 (logging context
and tracing) are already implemented, per [[2026-07-19-scripts-to-python-migration-design]] and
[[2026-07-19-logging-context-and-tracing-design]].

## Goal

Auto-generate every env file that derives from Terraform discovery, split per consumer, so
`make env-file` produces them all and no value is hand-maintained.

## Current state (verified by reading the code, 2026-07-20)

- A single root `.env` exists, generated by ~30 lines of `awk`/`printf` inside the Makefile's
  `env-file` target. It already uses an `# >>> AUTO-GENERATED … # <<< END` box that is
  regenerated while everything outside it is preserved (e.g. a hand-added
  `APIDOG_ACCESS_TOKEN`).
- **No service uses `env_file:`** — `docker-compose.yml` declares inline `environment:` lists:
  16 vars for `users`, 12 for `orders`.
- **Only 4 variables are `${...}`-interpolated inside the compose file itself**, and those are
  the only ones that genuinely require the ROOT `.env`: `COGNITO_USER_POOL_ID`,
  `COGNITO_CLIENT_ID`, `USERS_DB_PORT` (default `:-7001`), `ORDERS_DB_PORT` (default `:-7002`).
- The E2E suite loads the root `.env` via plain `dotenv` in `e2e/playwright.config.ts`
  (deliberately without `dotenv-expand`, because `API_GATEWAY_URL` contains a literal `$default`
  path segment that must not be expanded).
- `.env` and `.env.*` are already git-ignored.
- `APIDOG_ACCESS_TOKEN`/`APIDOG_PROJECT_ID` are referenced by no code — only preserved by the
  Makefile box.

> [!info] Correction that shaped this design
> The first assumption was that Compose can only read a file literally named `.env`. That is
> true ONLY for `${VAR}` interpolation inside the compose file. The `env_file:` service attribute
> accepts arbitrarily-named files and multiple files, injecting them into the container. Verified
> against Docker's documentation. This is what makes the per-service split viable.

> [!warning] Collision that forced a design change
> `DATABASE_WRITER_URL` and `DATABASE_READER_URL` exist in BOTH services with DIFFERENT values —
> and different formats: `postgres://test:test@floci:${USERS_DB_PORT}/users` for Users versus an
> ADO connection string `Server=floci;Port=${ORDERS_DB_PORT};Database=orders;` for Orders. A
> single shared `.env.<env>.services` file cannot hold two values for one key. The other 6 keys
> present in both services (AWS_*, GRPC_API_KEY, OTEL_*) do agree.

## Decisions

### 1. Per-service files, not a single `.services` file (deviation from the input, forced by data)

The original input asked for `.env.<environment>.services`. The DATABASE_*_URL collision makes
that impossible without renaming variables the application code already reads. Generated set:

| File | Contents | Consumer |
|---|---|---|
| `.env` | ONLY the 4 interpolated vars | compose `${...}` interpolation |
| `.env.local.infra` | Terraform outputs: Cognito ids, API GW url, DB endpoints, discovered ports | E2E, humans |
| `.env.local.users` | the full Users service environment | `env_file:` on the users service |
| `.env.local.orders` | the full Orders service environment | `env_file:` on the orders service |
| `.env.local.debug` | host-reachable connection strings (`localhost:<discovered port>`) | a SQL client; loaded by nothing |

Adding `tracking` / `events-pipeline` later means adding one more file, not editing a shared one.

### 2. `.env.example` is committed

Documents every key with an example value, so a fresh clone can see what it needs. `.env*` stays
git-ignored otherwise.

### 3. Compose migrates from inline `environment:` to `env_file:`

`users` and `orders` each load their own file. This removes ~28 lines of inline config and makes
the generated files the single source of truth rather than a duplicate of it.

### 4. Section boxes per file, custom values preserved

Each generated file carries `# >>> AUTO-GENERATED (<Service>)` and `# >>> CUSTOM (<Service>)`
boxes. Regeneration rewrites ONLY the auto-generated box; anything in CUSTOM survives — the same
mechanism the current `.env` already proves, applied per file.

### 5. The generator is Python, in `infra/scripts/`

Per [[scripting-language]] (block 1's convention). It replaces ~30 lines of awk/printf embedded
in the Makefile and reuses `lib3mrai` — which already provides the boto3 client factory and
per-engine port discovery. This is a concrete payoff of block 1: the shared package exists, so
the generator is small.

### 6. E2E loads infra + users explicitly

`playwright.config.ts` stops loading a monolithic `.env` and loads `.env.local.infra` plus
`.env.local.users`. Keep plain `dotenv` (no expansion) for the `$default` reason already
documented in that file.

### 7. Generation is wired into bootstrap, and validated

`make bootstrap` generates the env files BEFORE bringing services up, and the generator
validates that every required key for each service is present, failing with a clear message
naming the missing key. Rationale: moving to `env_file:` means a service that previously carried
its own config now depends on a generated artifact — an incomplete file must fail loudly at
generation time, not as a confusing runtime crash.

## Implementation shape — three layers

1. **Generator** — the Python script producing all five files, plus `.env.example`. Nothing
   consumes them yet, so this layer is independently verifiable by inspecting the output.
2. **Consumption** — compose switches to `env_file:`, E2E switches to the explicit files,
   `make bootstrap` wires generation in ahead of service start.
3. **Convention** — a vault note plus CLAUDE.md references: which file holds what, that
   generated files are never hand-edited, and that a new service adds a file.

Layer 2 is the risky one (a service failing to start); layer 1 delivers the files regardless.

## Verification

Per [[testing]], this block changes no HTTP endpoints. Acceptance:

- `make infra-down` then a full `make bootstrap` brings the whole stack up with services
  reading their generated env files.
- Gateway E2E passes with a real Cognito JWT, matching the current baseline.
- Regenerating preserves everything in the CUSTOM boxes.
- The generator fails loudly, naming the key, when a required value is missing.
- `.env.local.debug` connection strings actually connect from the host (they are the file's
  whole purpose, and nothing else exercises them).

## Risks / Open Points

- Migrating to `env_file:` changes how both services start; a missing or malformed file breaks
  startup where the compose file used to be self-contained. Mitigated by generating in bootstrap
  and validating.
- Compose gives `environment:` precedence over `env_file:`; if any inline entry is left behind it
  will silently win over the generated value.
- The four interpolated vars must stay in the root `.env` — moving them into a per-service file
  would break compose interpolation silently, since `${VAR}` with no value resolves to an empty
  string.
- `APIDOG_*` are consumed by nothing; the convention should say where such orphan values belong
  (CUSTOM box) rather than leaving them to accumulate.
