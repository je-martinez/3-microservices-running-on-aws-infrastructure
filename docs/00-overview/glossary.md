# Glossary

Definitions of key terms used across the **3MRAI** project. Each entry links to the note that owns the canonical definition or the ADR that establishes the decision. This glossary is a navigation aid — the linked notes are the source of truth.

---

## A

### Audit Fields

Standard metadata fields attached to every persisted entity to track its lifecycle:

| Field | Type | Purpose |
|---|---|---|
| `createdAt` | ISO-8601 timestamp | When the record was first created |
| `updatedAt` | ISO-8601 timestamp | When the record was last modified |
| `createdBy` | string (nano-id) | Identity of the actor who created the record |
| `updatedBy` | string (nano-id) | Identity of the actor who last modified the record |

All services must include these fields on every document. See [[audit-fields]] for the full convention.

---

## C

### CQRS (Command Query Responsibility Segregation)

An architectural pattern that separates the **write path** (commands that mutate state) from the **read path** (queries that return data). In 3MRAI:

- A service emits a domain event to an SQS queue (command).
- A Lambda handler consumes the event and writes to the DocumentDB **write replica**.
- The **read replica** replicates from the write replica and serves query handlers.

This separation allows write throughput and read latency to be optimised independently.

See [[cqrs]] for the implementation pattern and [[ADR-0002-cqrs]] for the architectural decision.

---

## D

### Dependency Injection (DI)

A design technique in which a component declares its dependencies as constructor parameters rather than instantiating them directly. All 3MRAI services use a DI container to wire infrastructure adapters (DB clients, SQS clients, gRPC stubs) to use-case and domain objects.

See [[dependency-injection]] and [[ADR-0008-screaming-arch-di]].

---

## F

### friendlyId

A human-readable, URL-safe identifier derived from an entity's name or title. Distinct from the internal [[#nano-id]] used as primary key. Used in URLs and display contexts where a meaningful slug is preferable to an opaque ID.

Example: an order with nano-id `ord_V1StGXR8_Z5jdHi6B-myT` might have `friendlyId: "order-for-jose-2026-06"`.

---

## I

### isDeleted

A boolean field (`isDeleted: true | false`) present on every persisted document, indicating whether the record has been logically removed. A document with `isDeleted: true` is excluded from normal query results but retained in the database for audit and recovery purposes.

Part of the [[soft-delete]] convention. See also [[ADR-0004-soft-delete-only]].

---

## F

### Floci

The **current** local AWS emulator for 3MRAI, superseding Ministack. An MIT-licensed emulator
exposing ~65 AWS services behind a single port (`:4566`, the same `AWS_ENDPOINT_URL` interface
as LocalStack). Terraform (`infra/environments/local`) and the services' AWS SDK clients target
it directly; ECS tasks and Lambdas run as real Docker containers on the compose network.

See [[ADR-0017-floci-local]] and the runbook [[local-dev-floci]].

---

## M

### Ministack

> [!warning] Superseded
> Superseded by **Floci** — see [[#Floci|the Floci entry above]] and [[ADR-0017-floci-local]].

The **former** local development environment for 3MRAI. A Docker Compose configuration that ran
all three microservices and local AWS emulators without requiring an active AWS account.

See [[ADR-0012-ministack-local]] and the (superseded) runbook [[local-dev-ministack]].

---

## N

### nano-id

A URL-friendly, collision-resistant random identifier generated by the [`nanoid`](https://github.com/ai/nanoid) library. In 3MRAI every entity's primary key is a **prefixed nano-id**: a short domain prefix followed by an underscore and a nano-id string.

Examples:
- User: `usr_V1StGXR8_Z5jdHi6B-myT`
- Order: `ord_abc123XYZ`
- Tracking event: `trk_def456ABC`

The prefix makes IDs self-describing and prevents accidental cross-domain usage.

See [[nano-id]] for the full convention and [[ADR-0005-nano-id-prefixed]] for the architectural decision.

---

## R

### Read/Write Replica

A DocumentDB topology where each service's cluster has two nodes with distinct roles:

| Role | Writer | Consumer |
|---|---|---|
| Write replica | Lambda CQRS handler | Receives all domain-event writes |
| Read replica | DocumentDB replication | Serves all service query handlers |

The replica split enforces the CQRS boundary at the infrastructure level and isolates write spikes from read latency.

See [[ADR-0006-read-write-replicas]] and [[cqrs]].

---

## S

### Screaming Architecture

A folder/module structure where the top-level names shout the **domain purpose** of the code, not the technical layer. Instead of `controllers/`, `services/`, `repositories/`, the root contains `users/`, `orders/`, `tracking/` — and within each domain the technical layers are nested.

The goal: open the codebase and immediately understand *what it does*, not *how it is structured*.

See [[screaming-architecture]] and [[ADR-0008-screaming-arch-di]].

---

### OpenObserve

A single-binary, open-source observability platform used in 3MRAI as the log-querying backend. OpenObserve ingests logs via OTLP from an OpenTelemetry collector, which in turn sources them from **Amazon CloudWatch** and, locally, Docker's fluentd driver. It supersedes SigNoz (see [[signoz-selfhost-migrator-blocker]] for why).

See [[openobserve-cloudwatch]] and [[ADR-0018-observability-openobserve]].

---

### Soft-Delete

The practice of never permanently deleting a database record. Instead, the record is marked with `isDeleted: true` and a `deletedAt` timestamp. All query handlers filter out soft-deleted records by default.

Benefits: auditability, recoverability, referential integrity for event logs.

See [[soft-delete]] for the full convention and [[ADR-0004-soft-delete-only]] for the architectural decision.
